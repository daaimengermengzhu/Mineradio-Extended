# Mineradio Extended 形态工坊实现计划

日期：2026-07-04

关联设计文档：

- `docs/superpowers/specs/2026-07-04-diy-sandbox-design.md`

## 目标

新增“形态工坊”，让用户可以用基础例子拼出自己的形态预设。

这里的形态预设只定义粒子结构和运动规则；歌曲封面、上传图片、文字和歌词只是底片素材，用来填充颜色、纹理和内容，不决定形态本身。

## 当前状态

已经完成：

- “用户存档”升级为“我的作品”。
- 视觉控制台新增“基础积木”面板。
- 现有面板可作为临时作品组合台继续保留。

需要纠偏：

- 当前“基础积木”更多是在现有预设上组合参数，不是真正的形态创建。
- 后续不能把“上传图片生成形态”作为主线。
- 新功能应从“形态预设”入口开始，而不是从“素材”入口开始。

## 阶段 1：入口和概念纠偏

### 要做什么

在现有“形态预设”区域增加“创建形态”入口。

建议 UI：

- 现有固定预设继续保留：封面粒子、唱片、星河等。
- 在预设列表里增加一个明显入口：`创建形态`。
- 点击后进入“形态工坊”面板。
- 当前“基础积木”面板文案调整为“作品组合台”或类似概念，避免和形态工坊混淆。

### 不做什么

- 不删除现有预设。
- 不移除当前“我的作品”。
- 不立刻重写 Three.js 粒子系统。

### 成功标准

- 用户能看到“创建形态”是独立入口。
- 用户能理解“形态”和“底片”是两件事。
- 老的视觉预设仍能正常点击和播放。

## 阶段 2：形态配方数据结构

### 新增数据模型

新增 `shapeRecipe`，用于描述形态规则。

示例：

```json
{
  "type": "mineradio-shape-preset",
  "version": 1,
  "name": "我的星环",
  "primitives": [
    {
      "id": "ring-1",
      "type": "ring",
      "count": 220,
      "radius": 1.2,
      "depth": 0.12,
      "rotation": 0.18,
      "audioFollow": 0.45
    },
    {
      "id": "spiral-1",
      "type": "spiral",
      "count": 360,
      "turns": 2.6,
      "radius": 1.5,
      "depth": 0.45,
      "audioFollow": 0.62
    }
  ],
  "materialBinding": {
    "mode": "cover-texture",
    "colorSource": "cover"
  },
  "camera": {
    "mode": "slow-orbit",
    "distance": 3.2
  }
}
```

### 存储

建议新增本地存储键：

- `mineradio-shape-presets-v1`

它只保存用户自定义形态预设，不混入歌曲播放记录、账号信息或音乐平台数据。

### 安全规则

- 不执行用户输入的 JavaScript。
- 不执行用户输入的 shader。
- 导入 JSON 时只接受白名单字段。
- 数值参数全部限制范围。
- 单个形态预设限制总粒子数量，第一阶段建议不超过 6000。

### 成功标准

- 可以保存、读取、复制、删除自定义形态预设。
- 无效 JSON 不会覆盖现有形态。
- 超出范围的参数会被钳制或拒绝。

## 阶段 3：基础例子库

### 第一批基础例子

基础例子是最小构件。

- 点：粒子簇。
- 线：粒子沿直线排布。
- 环：圆形轨道。
- 面：平面网格。
- 螺旋：按角度和半径旋转排布。
- 波：曲面起伏。
- 隧道：沿深度推进的环阵列。
- 卡片：带底片纹理的小平面。

### 每个例子的基础参数

通用参数：

- 数量
- 大小
- 透明度
- 深度
- 旋转
- 扩散
- 音频跟随强度

不同例子可以追加少量专属参数：

- 环：半径、厚度。
- 螺旋：圈数、收束程度。
- 波：振幅、频率。
- 隧道：层数、推进速度。
- 卡片：宽高比、朝向。

### 成功标准

- 用户能添加多个基础例子。
- 用户能调整每个例子的参数。
- 用户能删除、复制、排序基础例子。
- 预览会实时变化。

## 阶段 4：形态工坊 UI

### 布局

建议做成三栏或三段：

- 左侧：基础例子库。
- 中间：实时预览。
- 右侧：当前形态层级和参数。

如果空间不足，可以在当前视觉控制台内使用标签页：

- `例子`
- `结构`
- `参数`
- `保存`

### 主要操作

- 添加基础例子。
- 从基础结构开始创建。
- 选择某个例子并调参。
- 调整例子顺序。
- 保存为“我的形态预设”。
- 应用到当前播放画面。
- 导入 / 导出 `shapeRecipe` JSON。

### 文案原则

- 普通用户看到“点、线、环、波、卡片”，不直接看到 shader 等底层词。
- 高级用户可以看到 JSON，但 JSON 只是配方，不是代码。
- 底片入口只写“预览底片”或“填充素材”，避免写成“生成形态”。

## 阶段 5：渲染接入

### 第一阶段渲染策略

先不重写现有完整粒子系统，而是在现有 Three.js 场景中新增一个“自定义形态渲染器”。

建议函数边界：

- `normalizeShapeRecipe(raw)`
- `shapeRecipeToGeometry(recipe, materialContext)`
- `applyShapeRecipe(recipe)`
- `renderShapeWorkshopPreview(recipe)`
- `saveUserShapePreset(recipe)`

这里的“渲染器”不是新应用，只是把 `shapeRecipe` 转成 Three.js 能显示的几何体、粒子点和卡片。

### 材质和底片

底片只作为填充来源：

- 当前歌曲封面：取色、纹理、粒子采样。
- 上传图片：仅本地预览和填充。
- 文字 / 歌词：作为卡片文字或粒子文字层。

同一个形态预设切换歌曲后，应使用新歌曲封面重新填充。

### 成功标准

- 自定义形态能在主画面显示。
- 切换歌曲后，形态结构不变，底片内容更新。
- 性能不明显拖垮播放器。

## 阶段 6：我的形态预设

### 要做什么

在视觉预设区域新增“我的形态”分组。

用户保存的形态预设应能：

- 显示名称。
- 显示简短说明。
- 应用。
- 复制。
- 重命名。
- 删除。
- 导入和导出。

### 与“我的作品”的关系

- 我的形态：保存形态规则。
- 我的作品：保存形态 + 底片 + 歌词/镜头/颜色等完整快照。

这两个不要混为一个入口。

## 阶段 7：验证

### 代码检查

每阶段至少运行：

- `git diff --check`
- `node --check server.js`
- 前端内联脚本语法检查

### 浏览器验证

用本地页面验证：

- `http://127.0.0.1:3025/`

必须覆盖：

- 打开“创建形态”。
- 添加一个环。
- 添加一个螺旋。
- 调参数后预览变化。
- 保存为“我的形态预设”。
- 重新加载页面后仍能看到这个形态。
- 应用形态后播放画面能显示。
- 切换底片后形态结构不变。

### 风险验证

- 导入非法 JSON。
- 导入超大粒子数量。
- 删除当前正在使用的形态。
- 从旧版本 localStorage 升级。

## 风险和约束

- `public/index.html` 文件很大，第一阶段要把新逻辑集中放在清晰分区，避免散落修改。
- 不要一开始就支持用户写完整 shader，这会带来安全和性能风险。
- 不要把形态和底片再次混淆。
- 自定义形态第一阶段可以偏基础，但必须真的能由多个基础例子组合出来。
- 每个阶段都要保持现有音乐播放、登录和现有视觉预设可用。

## 建议提交拆分

1. `add shape workshop entry`
   - 只加入口和文案纠偏。

2. `add shape recipe storage`
   - 新增数据结构、读写、校验、导入导出。

3. `add shape primitive builder`
   - 新增基础例子库和 UI。

4. `render custom shape presets`
   - 接入 Three.js 预览和主画面渲染。

5. `save custom shape presets`
   - 我的形态预设列表、应用、复制、删除。

## 下一步

建议先实现第 1 步和第 2 步：

- 增加“创建形态”入口。
- 建立 `shapeRecipe` 的本地数据结构和校验。
- 暂时用静态预览占位，不急着接完整 Three.js 渲染。

这样可以先把产品概念和数据边界立住，再进入真正的粒子渲染。
