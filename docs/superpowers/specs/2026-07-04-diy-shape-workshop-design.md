# DIY 形态工坊：复用并拆解现有 3D 粒子预设设计

日期：2026-07-04

状态：已在对话中确认设计方向，等待实现计划。

## 背景

Mineradio-Extended 已经有一套真实运行中的 Three.js 3D 粒子视觉系统。现有形态预设包括 `emily专辑封面`、`滚筒`、`星球`、`虚空`、`唱片`、`星河`、`安魂`。

本轮二创 DIY 的目标不是重新做一个独立渲染器，也不是做一个和播放器效果割裂的静态编辑器，而是把原项目已有的 3D 粒子能力拆出来，让用户可以用更自由的方式组合、调整、保存，并在播放器里复现。

这里的“基础粒子”指点、线、环、面、螺旋、隧道、封面采样粒子、背景星尘等小型粒子单元；不是完整预设，也不是专辑封面、上传图片或文字本身。

## 当前代码观察

当前主要视觉逻辑集中在 `public/index.html`。

- `emily专辑封面`、`滚筒`、`星球`、`虚空`、`唱片`、`星河` 主要复用同一套主粒子系统，通过 shader 里的 `uPreset` 分支切换不同位置、颜色和律动算法。
- `安魂` 是独立的 `THREE.Points` 点云建模层，使用独立几何、材质、相机姿态和动画更新逻辑。
- 已有 DIY 形态数据雏形是 `shapeRecipe`，当前已经能表达基础粒子层、结构包、保存数据和简单工坊 UI。

因此第一阶段应采取“参数化拆解和复用”的策略，而不是强行一次性把 shader 完全拆成很多独立模块。

## 核心分层

### 基础粒子

基础粒子是最小的可拼装视觉单元。第一阶段包括：

- 点：粒子簇。
- 线：线性粒子排列。
- 环：圆环或轨道粒子。
- 面：封面采样平面或普通粒子平面。
- 螺旋：星河、旋臂、流动轨迹的基础。
- 隧道：深度推进的环阵列。
- 球面：星球类结构的基础。
- 星尘：星河背景和深度漂浮粒子。

基础粒子必须真实参与 3D 渲染，不能只是 DOM 或 2D 静态图形。

### 结构包

结构包是由多个基础粒子组成的半成品。第一阶段建议提供：

- 封面粒子结构：来自 `emily专辑封面`，重点是专辑封面采样面和基础起伏。
- 唱片结构：来自 `唱片`，拆成中心封面面、黑胶圆环、外圈粒子边缘。
- 星河结构：来自 `星河`，拆成螺旋粒子流、深度星尘、音乐律动。
- 隧道结构：来自 `滚筒`，拆成深度环阵和推进运动。
- 球面结构：来自 `星球`，拆成球面采样和自转。

用户点击结构包时，应叠加到当前形态里，不替换当前形态。

### 用户形态

用户形态是用户最终保存的一套完整组合。它由基础粒子和结构包组成，保存为 `shapeRecipe`。

用户形态保存后应出现在“我的形态”或对应入口里，能够在形态工坊继续编辑，也能够应用到主播放器。

### 底片素材

专辑封面、上传图片、文字都属于底片素材。底片可以被基础粒子采样、贴合或作为颜色来源，但它们不等于形态本身。

形态的核心仍然是粒子结构。

## 创作流程

第一阶段创作流程如下：

1. 用户点击“创建形态”，进入空白 3D 预览场景。
2. 用户添加基础粒子或结构包。每次添加都创建新层，不覆盖旧层。
3. 用户选中某一层，调整这一层的参数。
4. 用户保存形态。
5. 用户在播放器里应用该形态，播放器效果应与工坊预览一致。

第一阶段应优先保证流程完整和效果真实，再逐步增加更复杂的手柄拖拽、时间轴、节点编辑等能力。

## 数据结构

继续以 `shapeRecipe` 作为核心数据结构。它负责保存“用户想要什么”，不直接保存渲染细节。

示例结构：

```js
{
  name: '我的形态',
  primitives: [
    {
      type: 'ring',
      x: 0,
      y: 0,
      z: 0,
      size: 1.1,
      rotation: 0.3,
      count: 300,
      audioFollow: 0.4,
      opacity: 1,
      material: 'cover'
    }
  ],
  camera: {
    mode: 'slow-orbit',
    distance: 3.2
  }
}
```

字段含义：

- `name`：用户形态名称。
- `primitives`：基础粒子层列表。
- `type`：基础粒子类型。
- `x/y/z`：三维位置。
- `size`：整体大小。
- `rotation`：旋转参数。后续可以拆成 `rotationX/Y/Z`。
- `count`：粒子数量。
- `audioFollow`：跟随音乐律动的强度。
- `opacity`：透明度。
- `material`：素材绑定方式，例如封面、手动颜色、上传图片。
- `camera`：预览和播放器使用的相机策略。

## 渲染接入

新增或整理 `shapeRenderer` 这一层概念。它的职责是把 `shapeRecipe` 转换成真实 3D 粒子显示。

`shapeRenderer` 不应新造完全独立的视觉风格，而应复用现有能力：

- 主粒子系统的封面采样、颜色混合、律动参数、点纹理。
- `uPreset` 分支里的几何算法，例如平面、隧道、球面、唱片、星河。
- 现有音频数据，例如 bass、mid、treble、beat、energy。
- 现有相机、转场、粒子尺寸和 bloom 控制。

由于当前多个预设都集中在同一个 shader 中，第一阶段不要强行大拆 shader。更稳的做法是：

1. 先把每个预设能复用的算法整理成“基础粒子类型”和“结构包参数映射”。
2. 用 `shapeRecipe` 驱动这些参数。
3. 让形态工坊和主播放器共享同一个转换路径。
4. 稳定后再逐步把 shader 分支模块化。

## 第一阶段范围

第一阶段要完成：

- 基础粒子库：点、线、环、面、螺旋、隧道、球面、星尘。
- 结构包：封面粒子、唱片、星河、隧道、球面。
- 空白 3D 创建流程。
- 粒子层叠加，不互相覆盖。
- 每层可选中和参数调整。
- 用户形态保存、恢复、应用到播放器。
- 工坊预览和播放器效果使用同一套真实 3D 渲染逻辑。

第一阶段不做：

- 不开放 shader 代码编辑。
- 不开放任意 JavaScript 脚本。
- 不做完整节点编辑器。
- 不拆 `安魂` 的内部骨骼、牙齿、下颌等细节。
- 不重写整套 Three.js 渲染器。

## 安魂处理

`安魂` 目前是独立点云模型层，技术结构和普通封面粒子预设不同。第一阶段只把它作为完整结构包或参考预设保留。

后续如果要拆 `安魂`，应单独设计一版“模型点云拆解”方案，避免把普通粒子 DIY 和复杂模型编辑混在一起。

## 验收标准

实现后需要满足：

1. 原有预设仍然正常展示，没有明显视觉退化。
2. 工坊预览和播放器应用后的效果一致。
3. 基础粒子是真实 3D 粒子，不是 2D 占位图。
4. 添加基础粒子或结构包是叠加行为，不是替换行为。
5. 保存后刷新页面或重启应用，用户形态仍可恢复。
6. 参数调整能真实影响 3D 效果，例如位置、深度、旋转、粒子数量、音乐跟随。
7. 如果某个旧预设暂时只能作为完整结构包复用，应在 UI 上避免暗示它已经能完全拆解。

## 风险和约束

- `public/index.html` 已经很大，实施时应避免无关重构。
- shader 分支集中度高，拆解时要先做小范围适配，不要一次性重写。
- 粒子数量需要上限，避免用户组合过多导致卡顿。
- 保存格式需要版本号，后续升级时才能兼容旧用户形态。
- 工坊和播放器必须走同一条渲染路径，否则用户会认为“预览骗人”。

## 推荐实施顺序

1. 梳理现有预设到基础粒子的映射表。
2. 扩展 `shapeRecipe`，补齐 `z`、透明度、素材绑定、相机和版本字段。
3. 做真实 3D 工坊预览，不再只依赖 2D 静态舞台。
4. 接入结构包叠加和每层参数调整。
5. 保存用户形态并在播放器里应用。
6. 用原有预设做回归检查，确认旧效果没有被破坏。
